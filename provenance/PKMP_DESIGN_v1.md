AI and Human collaborators in a form that:- survives across sessions, agent instances, and tool changes;- is canonical (one source of truth, not a copy synchronized across many places);- can be projected into any number of human-facing views (Markdown documentation, HTML wikis, PDF reports, OpenSpec definitions) without those views ever becoming the truth;- treats Knowledge — not files, not documents, not conversations — as the primary deliverable.Concretely, PKMP provides:1. **A canonical Information Model** that describes the kinds of knowledge a project contains (decisions, requirements, designs, experiments, research, etc.) and the relationships among them.2. **A Repository structure** that materializes the Information Model on disk in a way both AI and Human can navigate.3. **A Registry** by which the Repository describes itself, so that any agent can discover its own contents without out-of-band documentation.4. **A Capability model** for the executable parts of the system (renderers, validators, extractors), where capabilities are exchangeable rather than baked in.5. **A Knowledge Lifecycle** (Research → Decision → Implementation → Operation → Feedback → Research) that all knowledge objects follow.6. **A Publishing Pipeline** that turns Conversation into Knowledge Candidates, Knowledge Candidates into Draft Packages, and Draft Packages into Published Knowledge.### §1.2 What PKMP Is NotPKMP is **not** any of the following, despite resemblances:**Not a Documentation Platform.**Documentation Platforms (MkDocs, Docusaurus, GitBook, Notion, Confluence, etc.) treat Markdown or rich-text documents as the primary artifact. PKMP treats Markdown as a *view* — a rendered projection of the underlying Model. The Model, not the Markdown, is canonical. Documentation is a downstream consequence of Knowledge, not a substitute for it.**Not a File-Centric Knowledge Base.**File-centric tools (Obsidian, Logseq, plain Git Markdown repositories) treat the file tree as authoritative. PKMP treats the file tree as a generated projection of the Information Model and Registry. Two PKMP Repositories with identical Knowledge content can have different file layouts, both valid, both regenerable.**Not a Project Management System.**PKMP does not track tasks, sprints, tickets, or velocity. Project management tools may *consume* PKMP Knowledge (for example, by reading Decision Records) and may *contribute back* via Knowledge Candidates, but they are external to PKMP.**Not an Operating System.**The earlier name "Project Operating System (POS)" was rejected because "Operating" implies runtime orchestration of processes. PKMP does not orchestrate processes; it models the knowledge that processes consume and produce. (See §1.3, §12.)**Not AXIS.**PKMP overlaps with several AXIS concepts (notably UDR and the Document/Record separation) but is intentionally an independent project. See §1.5 and Appendix C (DC-001).**Not a Plugin Platform.**PKMP uses the term **Capability** rather than **Plugin** deliberately. Plugin connotes a runtime-bound implementation. Capability connotes an abstract responsibility that may have many exchangeable implementations. (See §9.)### §1.3 Project HistoryPKMP arose from a chain of project re-scopings. The history is recorded here because the chain explains why PKMP is structured as it is and why several apparently natural alternatives were rejected.**Stage 1 — Windows Server / Docker / Minecraft.**The originating problem was operational: how to run small, secure server workloads (initially exemplified by a Minecraft server on Docker on Windows Server). This is not PKMP's problem; it became the originating use case for SRM (see Stage 3).**Stage 2 — SRM (Secure Runtime Manager).**The operational problem was generalized to a Secure Runtime Manager. SRM is intended to manage container workloads with security guarantees. SRM is a *real product* with its own requirements and architecture. SRM is not PKMP.**Stage 3 — The realization.**Designing SRM revealed that SRM's long-term maintainability required a substrate on which Architecture Decisions, Requirements, Designs, and Operational Records could be modeled in a form an AI agent could maintain across sessions. No existing platform provided this. The substrate became a project in its own right.**Stage 4 — POS (Project Operating System).**The substrate was initially named "Project Operating System". This name was rejected because the substrate does not Operate projects; it Models them. (See §12.)**Stage 5 — PKMP (Project Knowledge Modeling Platform).**The renaming reflects the actual purpose. The platform's job is to model Project Knowledge such that AI and Human can both work with it. POS is now treated as one Capability under PKMP (specifically, the operational/runtime aspect, if and when it is needed), not as the platform itself.**Stage 6 — Current.**PKMP is the priority deliverable. SRM is repositioned as the **first Reference Project** that runs on PKMP. SRM validates that PKMP works in practice; PKMP enables SRM to be maintainable over years.### §1.4 SRM as First Reference ProjectSRM serves a specific role in PKMP development: it is the **first non-trivial project to live entirely inside PKMP**.This positioning has consequences for both projects:- **For PKMP**: SRM is the empirical validator. Any PKMP design decision that cannot be applied to SRM's actual artifacts (its Architecture Decisions, its component Designs, its operational Runbooks) indicates a gap in PKMP, not a problem with SRM. PKMP development must remain ahead of SRM's needs by a small margin — enough to be usable, not so much that it speculates beyond what SRM demonstrably requires.- **For SRM**: SRM accepts that its own development is downstream of PKMP. SRM does not start serious implementation until PKMP is at least at the level of Phase 1 of the Bootstrap Plan (see §11). SRM does not maintain its own out-of-band documentation system; everything lives in PKMP from the start..SRM is not the only future Reference Project. Other candidates include:- a home-server management project;- future projects that emerge under the same Human Knowledge Owner.PKMP is designed to host multiple Reference Projects in the long run, not to be SRM-shaped.### §1.5 Relationship to AXISPKMP and AXIS (the framework comprising ADSD, SDT, DDU, CbS, UDR, VCS, etc.) share several concepts. Most notably:- Both have a **Decision Record** primitive (PKMP's Decision Record / AXIS's UDR).- Both separate **Document** from **Record** (PKMP explicitly in §3.8; AXIS implicitly through UDR vs. design docs).- Both treat **Capability exchangeability** as central (PKMP §9; AXIS's various pluggable components).- Both insist on a Single Source of Truth (PKMP's Canonical Source hierarchy; AXIS's SPEC/CbS).These overlaps are not accidental: PKMP and AXIS arose from the same underlying problem (long-term project knowledge maintenance under AI-augmented development) and the same Human Knowledge Owner has informed both.**The official position is recorded as Decision Candidate DC-001 (Appendix C):**- **Short term**: PKMP is designed and implemented **independently** of AXIS.. PKMP does not import AXIS terminology, file layouts, or implementation. AXIS-equivalent concepts (e.g., UDR) coexist with PKMP-equivalent concepts (e.g., Decision Record) as separate implementations until further notice.- **Long term**: When PKMP is stable (post v1.0 Published), the relationship is re-evaluated. Possible outcomes include eventual merger, formal interop specifications, or continued independence. The choice is itself a Decision to be recorded then, not now.**Rationale for independence** (summarized; see DC-001 for full):- PKMP is intended as a *Meta Layer* on which AXIS itself could eventually be modeled. A Meta Layer cannot be a subordinate of one of its tenants.- AXIS is a mature framework in active use; forcing PKMP to conform to AXIS terminology would compromise PKMP's design integrity.- SRM is an independent Reference Project, deliberately chosen to *not* be an AXIS use case, so that PKMP can be validated without AXIS coloring the result.**Practical consequence for this document:**- PKMP terminology takes precedence over AXIS terminology in this document.- AXIS concepts are referenced for orientation in Appendix B (Mapping to AXIS Concepts) but are never normative.- The CLI MUST NOT import AXIS file layouts, naming conventions, or registry schemas when generating the PKMP Repository.---## §2 — Non-Negotiable DecisionsThis section restates the eleven Non-Negotiable Decisions from `provenance/PKMP_PROJECT_KNOWLEDGE_v1.0.md`. Read it before reading §4 onward. Every later section uses these terms as defined here.The section is organized bottom-up by abstraction level: §3.1 introduces the most abstract concept (Knowledge), §3.2–§3.6 walk down the hierarchy to the most concrete (View), §3.7 ratifies the hierarchy formally, and §3.8–§3.11 introduce the orthogonal distinctions (Document vs. Record, Capability, Registry, Knowledge Type System) that cut across the hierarchy.### §3.1 Knowledge**Knowledge** is the structured representation of decisions, requirements, designs, observations, experiments, and judgments that constitute a project's truth.Knowledge is what PKMP exists to model. By Decision-1, Knowledge is the primary deliverable.**Properties of Knowledge in PKMP:**| Property | Meaning ||---|---|| **Persistent** | Knowledge outlives sessions, agents, tooling. || **Canonical** | Knowledge is the truth; everything else is derivation (Decision-2). || **Typed** | Knowledge has Knowledge Types (see §3.11). || **Stateful** | Knowledge progresses through lifecycle states (Draft → Reviewed → Published, see §5.2). || **Decomposable** | Knowledge is made up of typed sub-units (Decisions, Designs, Observations, etc.) related by typed relations. || **Reconstructable** | Knowledge can be regenerated from its canonical representation; no out-of-band context is required (Decision-1 + Success Criteria). |**What Knowledge is not:**- Knowledge is not a file. Files are storage.- Knowledge is not a document. Documents are projections (Decision-4).- Knowledge is not a conversation. Conversations are inputs to Knowledge (Decision-7 + §6.5).- Knowledge is not source code. Source code may *implement* Knowledge (e.g., a Capability implementation), but the code is not the Knowledge.### §3.2 Information ModelThe **Information Model** is the structured description, internal to a PKMP-managed project, of:- the **Knowledge Types** the project recognizes;- the **attributes** that each Knowledge Type carries;- the **relations** that Knowledge Types may have to one another;- the **constraints** that constrain valid instances.By Decision-2, the Information Model is the Canonical Source.The Information Model is itself a piece of Knowledge — it is the Knowledge that describes the project's Knowledge. This recursion is intentional and is how PKMP bootstraps (see §11).**Information Model vs. Schema vs. Model:**These three terms are easily confused. Section §3.7 formalizes the relationship; for now:- **Information Model**: the *conceptual* structure of the project's Knowledge (e.g., "this project recognizes Decisions, Requirements, and Designs, with these relations among them").- **Schema**: the *machine-checkable* expression of the Information Model in a constraint language (see §3.3).- **Model**: a *concrete instance* of Knowledge conforming to a Schema (see §3.4).The Information Model is more abstract than any single Schema. A Schema is one syntactic encoding of (part of) the Information Model.### §3.3 SchemaA **Schema** is the constraint language expression of (part of) the Information Model.Schemas in PKMP are:- written in **JSON Schema-compatible YAML**;- the validation contract that every Model must satisfy;- themselves Knowledge (they describe Knowledge Types).**Schemas serve four consumers:**1. **Validators** check that Models conform to Schemas. A non-conforming Model is rejected before it enters the Repository.2. **Renderers** read Schemas to know what to do with each Knowledge Type when generating Views.3. **Editors** (VSCode, Zed, Cursor, future PKMP-aware editors) use Schemas for autocomplete, hovers, and inline validation while a Human or AI edits Models.4. **CLIs** use Schemas to generate scaffolding for new Models.**Schemas are not documentation.** A Schema's role is type definition and validation, not human-readable explanation. Where a Knowledge Type needs explanation, that explanation lives in a Document (see §3.8); the Schema only carries enough description to drive validation and editor tooling.### §3.4 Model (YAML)A **Model** is a concrete instance of Knowledge.Models in PKMP are:- written in **YAML** (the canonical Model representation format);- required to **conform to a Schema** — Models without Schemas are rejected;- the **payload** that Renderers consume to produce Views (§3.5–§3.6).**Why YAML:**YAML is chosen as the canonical Model representation for the following reasons:1. **Machine-readable**: every modern language has a YAML parser.2. **Human-readable enough** for review when needed (though the human-facing form is normally a rendered View, not the raw YAML).3. **Schema-validatable** through JSON Schema (YAML→JSON is structurally lossless for PKMP's intended use).4. **AI-friendly**: LLMs handle YAML reliably; ambiguity in parsing is minimal.**JSON is not the canonical format.** JSON is a valid output of YAML serialization and may be produced by Renderers, but the *source* form that a Human or AI authors is YAML.**Markdown is not the canonical format** (Decision-4). Markdown is a View output of a Renderer that consumes a Model.### §3.5 RendererA **Renderer** is a **Capability** (see §3.9) that consumes Models and emits Views.Renderer signature, abstractly: `Renderer(Model, RenderOptions) → View`**Examples of Renderers PKMP expects to support:**- **Markdown Renderer** — produces `.md` files for human navigation in the Repository.- **HTML Renderer** — produces standalone HTML for web rendering.- **PDF Renderer** — produces PDF for archival or sharing.- **OpenSpec Renderer** — produces OpenSpec output where required (see Decision-5 / §9.5).- **Wiki Renderer** — produces wiki-flavored Markdown variants for systems that require them.- **(Future)** **Editor Extension Renderers** — VSCode / Zed / Cursor extensions that present Models as rich editable Views inline.**Renderers are exchangeable.** No part of PKMP Core assumes a specific Renderer. The Markdown Renderer is the most operationally important (because the Repository's human-facing surface is mostly Markdown), but it is replaceable: a future better Renderer can be substituted without disturbing the Information Model or the Schemas.**Renderers are not bidirectional.** A Renderer goes Model → View. The reverse direction (View → Model) is not a Renderer's responsibility; it requires a different Capability (typically an Extractor — see §10.3 for Knowledge Extractor's pending status).### §3.6 ViewA **View** is the human-facing form of Knowledge produced by a Renderer.**Views are non-canonical** (Decision-4). Editing a View does not edit Knowledge. The View can be regenerated at any time from the Model, and that regeneration discards any direct edits to the View.**Common Views in PKMP:**- Markdown documents (the dominant View in a generated Repository)- HTML pages- PDF documents- OpenSpec definitions (when produced by the OpenSpec Renderer)- Wiki pages- (Future) interactive editor surfaces**View-vs-Model discipline:**A frequent failure mode in document-centric systems is for someone to edit the rendered Markdown and expect the change to "stick." In PKMP this does not work, and is not supposed to work. The discipline is:- To change Knowledge, edit the Model.- If a View is wrong, either the Model is wrong (fix the Model) or the Renderer is wrong (fix the Renderer).- A View should not be edited directly except for ephemeral, throwaway purposes that will not be regenerated.This discipline is structurally enforced by the Knowledge Publishing Pipeline (§6): Drafts flow Model-first, and Reviews are conducted against the Snapshot as a whole, not against rendered-text diffs in isolation.### §3.7 Canonical Source Hierarchy (Final)This is the ratified Canonical Source hierarchy for PKMP. It is final and binding.```Knowledge   ↓  (canonically represented as)Information Model   ↓  (constrained by)Schema   ↓  (instantiated by)Model (YAML)   ↓  (consumed by)Renderer (Capability)   ↓  (produces)View (Markdown / HTML / PDF / OpenSpec / Wiki / …)```**Reading the hierarchy:**- **Knowledge** is the conceptual primary deliverable (Decision-1). It is the *what*, not the *how*.- **Information Model** is the canonical *representation* of Knowledge (Decision-2). It is what gives Knowledge structure machines can act on.- **Schema** is the constraint language for the Information Model. Schemas make the Information Model machine-checkable.- **Model** is a concrete YAML instance that conforms to a Schema. Models hold the actual project content.- **Renderer** is a Capability that consumes Models and produces Views.- **View** is what a Human sees. Views are never canonical.**Resolution of source-document variation** (closing Open Issue #1):The earlier internal variations are reconciled as follows:| Source statement | Interpretation under §3.7 ||---|---|| Decision-2: "Information Model is the Canonical Source." | Among representations, the Information Model is the canonical one. The CLI MUST NOT treat Markdown, HTML, PDF, or any other View as authoritative under any circumstance.### §3.8 Document vs. RecordPKMP separates **Documents** from **Records**. This separation is non-negotiable and is one of the most important distinctions in the platform.**Document.** A Document is a deliverable produced *for human explanation*. Documents present conclusions, decisions, and designs in a form intended to be read and understood by people who were not party to the underlying work.Examples: Architecture Document, Security Document, Project Document, Operational Runbook, Design Document.**Record.** A Record is the *evidence trail* for the work that produced the Documents. Records preserve facts, observations, raw decisions, experimental results, meeting notes, and other primary material.Examples: Decision Record (the PKMP-native form of UDR), Research Record, Experiment Record, Benchmark Record, Meeting Record, Investigation Record, PoC Record.**Key distinctions:**| Aspect | Document | Record ||---|---|---|| Reader | Anyone who needs to understand | Someone investigating the trail || Content | Conclusion | Fact || Mutability | Edited as understanding evolves | Immutable once Published || Tone | Explanatory, possibly persuasive | Neutral, evidentiary || Reference direction | Documents reference Records | Records do not reference Documents (in general) |**Why this matters in PKMP:**- A common failure mode is to embed rationale inline in a Document. This makes the Document large, hard to maintain, and tangled with history. In PKMP, the Document says *what was decided*; the Record says *why it was decided* and *what alternatives were rejected*. The Document references the Record.- Records are append-only by design. Documents are regenerable.- The Knowledge Publishing Pipeline (§6) operates differently on Documents and Records. Records, once Published, cannot be retroactively edited; Documents are regenerable views over current state.**Mapping to Repository structure:**The Repository will (subject to §7) typically split Documents and Records into separate top-level directories (commonly `docs/` and `records/`). The exact layout is determined in §7. The separation, however, is architectural and does not depend on directory naming.### §3.9 CapabilityA **Capability** is an executable component that PKMP uses but does not architecturally depend on.By Decision-5 (and §9), Capabilities are exchangeable: any specific Capability implementation may be replaced without disturbing the Information Model, the Schemas, the Registry, or the Models.**Examples of Capabilities:**- **Renderers** (Markdown, HTML, PDF, OpenSpec, Wiki, …)- **Validators** (Schema validators, cross-reference validators)- **Extractors** (e.g., the proposed but not-yet-ratified Knowledge Extractor for Conversation → Knowledge Candidate; see §10.3)- **Installers / Generators** (bootstrap tooling)- **External agents** (GitHub API integration, future AI agents)- **OpenSpec itself** as a Capability description format (Decision-5)**Why "Capability" rather than "Plugin":**The term "Plugin" was considered and rejected. "Plugin" connotes a specific runtime binding (a dynamically-loaded library, a Webpack-style plugin entry point) and implies an implementation host. "Capability" is more abstract: it names the *responsibility* without committing to how that responsibility is fulfilled. A Capability can be implemented as a script, a binary, an external service, an AI agent, or even a human procedure.The CLI MUST treat Capabilities as exchangeable. The CLI MUST NOT hardcode a specific Renderer or Validator into the Repository's structural integrity.See §9 for the full Capability Model.### §3.10 RegistryThe **Registry** is the metadata layer by which the Repository describes itself.The Registry is *not* a Database. It is *not* a search index. It is a set of YAML files that record:- the locations of Documents, Records, Schemas, Models, Capabilities, Workflows, and Lifecycle definitions within the Repository;- the relationships among them;- the Capabilities available to operate on them;- the conventions the Repository follows.**Key consequence: Self-Describing Repository.**A PKMP Repository carries enough Registry information to be entirely self-explanatory to any agent (AI, CLI, Human, validator) that reads it. No out-of-band documentation is required to understand a PKMP Repository's structure.**Registry, Schema, and Model relationship:**- The **Schema** says what *kinds* of things exist in a Repository (Knowledge Types).- The **Model** is a concrete instance of one of those kinds.- The **Registry** says where those concrete instances live in the Repository's file layout.Schema is type-level; Model is instance-level; Registry is locator-level.See §8 for the full Registry Specification.### §3.11 Knowledge Type SystemPKMP defines a **Knowledge Type System**: the set of categories that a piece of Knowledge can belong to, the attributes each category carries, and the relations that may hold between categories.**Top-level Knowledge Types:**| Type | Role | Example ||---|---|---|| **Document** | Human-facing deliverable | Architecture Document || **Record** | Evidence trail | Decision Record || **Schema** | Type/constraint definition | Decision Record Schema || **Model** | Concrete Knowledge instance | A specific Decision Record YAML || **Capability** | Executable component | Markdown Renderer || **Workflow** | Process definition | Knowledge Publishing Workflow || **Lifecycle** | State machine definition | Knowledge Lifecycle || **Registry** | Repository self-description | Repository Registry || **View** | Rendered output | Generated Markdown documentation |This list is not closed: a PKMP-managed project may define **project-specific Knowledge Types** by extending the Information Model. The CLI MUST NOT define new top-level Knowledge Types on its own (Decision-11); new types arise from Human-ratified design decisions captured as Decision Records.The Knowledge Type System is itself a piece of Knowledge — it is defined by Schemas, which are themselves Models. This recursion is intentional. See §11 for how the recursion is bootstrapped.---## §4 — Roles and ResponsibilitiesPKMP recognizes three roles. Decisions-9, -10, and -11 establish them. This section formalizes what each role does, what each role does not do, and where the boundaries lie.### §4.1 Human as Knowledge OwnerThe Human is the **Knowledge Owner**.**The Knowledge Owner:**- **Reviews** Draft Packages produced by the AI/CLI.- **Decides Publish**: approves or rejects Drafts for Publication. No Publish event occurs without an explicit Human decision.- **Sets long-term direction**: scope, priorities, strategic choices.- **Arbitrates** when the AI presents competing design candidates and cannot itself decide.- **Defines what counts as Knowledge**: ultimately, the Human's judgment governs what becomes Published Project Knowledge.**The Knowledge Owner does not:**- write raw Knowledge artifacts directly (the AI does that);- operate the CLI's mechanical steps (the CLI does that);- maintain the Information Model in detail (the AI does that under design constraints set by the Human).**Why:**The division of labor reflects comparative advantage. The AI is faster at production and consistency-checking. The Human is irreplaceable at judgment, taste, and strategic priorities. PKMP's operating model concentrates the Human's time on the activities where the Human's judgment is required.### §4.2 AI as Chief ArchitectThe AI is the **Chief Architect**.**The Chief Architect:**- **Maintains overall integrity** of the Information Model, Schemas, Registry, and Repository.- **Produces Draft Packages** for Human Review.- **Extracts Knowledge Candidates** from Conversation (with the Human's framing).- **Designs new structures** when existing structures cannot express a need.- **Proposes Decision Records** for any non-trivial choice, including the rejected alternatives.- **Defers to the Human** on Publish decisions and on disputes the AI cannot itself resolve.**The Chief Architect does not:**- have persistent access to the Repository filesystem (the CLI does);- Publish Knowledge unilaterally (the Human does);- introduce new architectural concepts silently. New concepts must be proposed as Decision Records or Decision Candidates (Decision-11 applies to the AI also, by mirror).**Practical note on AI identity:**"AI" in PKMP refers to any session-scoped reasoning agent operating in the Chief Architect role. Different AI instances (different chat sessions, different models, different days) may all play Chief Architect in different sessions. Persistence across sessions is the Repository's responsibility, not the AI's memory's. This is the practical consequence of "Conversation history SHALL NEVER be required for reconstruction" (PKMP_PROJECT_KNOWLEDGE_v1.0, Knowledge Preservation).### §4.3 CLI as Implementation AgentThe CLI is the **Implementation Agent**.**The CLI:**- **Generates Repositories** from the Information Model + Schemas + Registry..- **Expands specifications** into complete artifacts (full Schemas from outlines, full Validators from contracts, full Renderers from interface descriptions).- **Executes deterministic algorithms**: validation, rendering, registry resolution.- **Refactors** code without changing the architecture.- **Surfaces gaps**: when the design is silent or contradictory, the CLI reports the gap rather than guessing.**The CLI does not:**- introduce new architectural concepts (Decision-11);- silently rename canonical identifiers (Decision-11);- discard provenance (Decision-11);- make decisions that the design leaves Open (the CLI surfaces them, the Human decides);- treat its own preferences as authority.The full CLI Working Agreement is in §14.### §4.4 Role BoundariesThe three roles correspond to three distinct authorities:| Authority | Role | Examples ||---|---|---|| **Design** | AI (Chief Architect) | Proposing the Information Model; structuring the Repository || **Implementation** | CLI (Implementation Agent) | Generating files; running validators; executing renderers || **Publishing** | Human (Knowledge Owner) | Approving Drafts; setting direction; resolving disputes |**Boundary rules:**1. The AI does not Publish; the AI Drafts. The Human Publishes.2. The CLI does not Design; the CLI Implements against ratified design.3. The Human does not write Drafts; the Human reviews them.**Edge cases:**- *A Human writes a quick note in a Markdown file.* This is not a Knowledge contribution to PKMP. It is either an ephemeral comment (will be discarded on next regeneration) or a Knowledge Candidate (must enter the Pipeline via §6).- *The AI writes implementation code.* This is acceptable when the AI is acting as a coding agent (e.g., during initial bootstrap or in a non-PKMP context). In PKMP's operating model, the CLI normally writes implementation code under the AI's design. Code-writing AI work is treated as implementation, not design.- *The CLI produces a Decision Record.* Acceptable if the Decision Record was Drafted by the AI and the CLI is materializing it on disk. Not acceptable if the CLI authored the Decision content itself.---## §5 — Knowledge LifecycleThis section formalizes the lifecycle through which Knowledge passes in PKMP.. The lifecycle was originally discussed under the name "Loop Engineering"; it has been renamed to **Knowledge Lifecycle** to make the central insight explicit: it is *Knowledge* that cycles, not loops in isolation.### §5.1 OverviewKnowledge in PKMP follows a recurrent cycle:```        Research           ↓       Experiment           ↓        Decision           ↓     Knowledge Model           ↓          View           ↓     Implementation           ↓        Operation           ↓        Feedback           ↓        Research  ← (cycle continues)```Each stage produces or refines a Knowledge object that becomes input to the next stage. The cycle is the normal mode; there is no "end state." Knowledge that ceases to cycle is dormant Knowledge — still preserved, but not under active development.This is a conceptual lifecycle. It describes how Knowledge evolves over a project's lifetime. It is **distinct from** Knowledge States (§5.2), which describe a single Knowledge object's progression from Draft to Published.### §5.2 Knowledge StatesA given Knowledge object — say, a particular Decision Record — passes through three states:```   Draft  →  Reviewed  →  Published```| State | Meaning | Producer | Consumer ||---|---|---|---|| **Draft** | The AI's proposed Knowledge artifact, not yet Reviewed. | AI (via CLI) | Human (for Review) || **Reviewed** | The Human has reviewed and either approved as-is or annotated with required changes. | Human | AI (for revision) or CLI (for Publishing) || **Published** | Officially part of Project Knowledge. Supersedes prior Drafts of the same Knowledge (Decision-8). | Human (decision) + CLI (materialization) | All future readers and processes |**Conversation is not a state.** Conversation is *prior to* Draft. Conversation feeds Knowledge Candidates (§5.3) which become Drafts.**State transitions are gated:**- **Draft → Reviewed**: requires Human Review (no automatic promotion).- **Reviewed → Published**: requires Human Publish decision (no automatic promotion).- **Published → (revision)**: a Published artifact is not edited in place. Changes produce a new Draft that supersedes the prior Published version (Decision-8); the prior Published version remains in history.### §5.3 Knowledge CandidateA **Knowledge Candidate** is the seed form of Knowledge: a unit identified during Conversation (or other exploratory work) as potentially worth preserving, but not yet shaped into a Draft.Knowledge Candidates exist to enforce the discipline that **nothing goes directly from Conversation to Knowledge**. The AI does not promote Conversation content straight to Project Knowledge. Conversation produces Candidates; Candidates are reviewed (lightly) and shaped into Drafts; Drafts undergo full Human Review; Reviewed Drafts become Published.**A Knowledge Candidate is characterized by:**- It was identified during an Architecture Session or other exploratory work..- It has not yet been shaped into a Schema-conformant Model.- It carries enough information to recover the intent if revisited later (Information Loss Zero, §5.6).- It is held in the Draft Package's working area, not in `records/` or `docs/`.**Three Candidate sub-categories** are distinguished:- **Knowledge Candidate** (general): any content identified for preservation..- **Decision Candidate** (§5.4): a candidate that, if Published, will become a Decision Record.- **Action Candidate** (§5.5): a candidate that, if Published, will become an Action item for downstream work.### §5.4 Decision CandidateA **Decision Candidate** is a Knowledge Candidate that proposes a Decision.A Decision Candidate carries the same shape as a Decision Record (context, decision, alternatives considered, rationale, consequences) but has not been Published. The Human reviews the Decision Candidate; upon approval it becomes a Published Decision Record.**Why distinguish:**- Decision Records, once Published, are immutable (§3.8). The Decision Candidate stage exists so that Decisions are deliberated and shaped before they become irrevocable.- Decision Candidates surface the alternatives the AI considered. The Human can challenge the choice or request additional alternatives before Publishing..**Notation in this document:**Decision Candidates are tagged `DC-NNN` in Appendix C. DC-001 (PKMP–AXIS relationship) is the first; further Decision Candidates accumulate during PKMP's own development.### §5.5 Action CandidateAn **Action Candidate** is a Knowledge Candidate that proposes an Action: something that needs to happen next.Actions are not TODOs in a task tracker. Actions are Knowledge: a next step justified by, and traceable to, the work that produced them. Action Candidates carry:- the trigger (what prompted the Action);- the proposed Action;- the expected outcome;- the linkage back to the Knowledge that motivated it.On Publishing, an Action Candidate may be exported to a task tracker as a TODO, but the canonical record of the Action remains in PKMP. The task tracker entry is a View.**Why this matters:**A frequent failure mode is for design conversations to produce loose TODOs that lose their context as soon as they enter a task tracker. By treating Actions as Knowledge first, PKMP preserves the context — the *why* behind the *what*.### §5.6 Information Loss Zero Principle**Information Loss Zero** is the principle that *no information generated by a deliberate process is summarized away*.Concretely:1. **Conversation → Knowledge Candidate**: the AI does not "summarize" Conversation. The AI *Compiles* Conversation into Knowledge Candidates, preserving substance, decisions, alternatives, and rationale. Where Conversation contains ambiguity, the AI surfaces the ambiguity in the Candidate (e.g., as an Open Question or Decision Candidate) rather than picking one interpretation silently.2. **Knowledge Candidate → Draft**: the Human and AI shape Candidates into Drafts. Reshaping is allowed; meaning-change is not. Where a Draft must depart from a Candidate's content, the departure is recorded (typically as a Decision Record or change note).3. **Draft → Published**: Publishing does not summarize. The Reviewed Draft *is* what gets Published. Edits during Review are recorded; the Reviewed-then-Published artifact carries the trail.4. **Published → Documentation View**: even rendered Documents are not summaries of Records in the loss-zero sense — they are projections that may omit detail for readability, but the underlying Record remains intact and is referenced (§3.8).**Operational consequence for the AI:**When in doubt, *preserve more than feels necessary*. The downstream cost of preserving an apparently-extra detail is small (it lives in a Record); the cost of having silently discarded it is high (the rationale is lost).**Operational consequence for the CLI:**The CLI MUST NOT compress, summarize, or otherwise reduce Knowledge content during materialization. The CLI may *render* Knowledge into shorter Views (that is a Renderer's job and is reversible), but the underlying Knowledge must remain intact.---## §6 — Knowledge Publishing PipelineThis section formalizes how Knowledge moves from Conversation to Published Project Knowledge.### §6.1 Overview```   Conversation        ↓   Knowledge Candidate        ↓   Draft Package        ↓   Human Review        ↓   Published Project Knowledge```Each arrow is a deliberate transition:| Transition | Performed by | Output ||---|---|---|| Conversation → Knowledge Candidate | AI (Chief Architect) | Candidates added to Draft Package working area || Knowledge Candidate → Draft Package | AI + CLI | Repository Snapshot + REVIEW + CHANGELOG || Draft Package → Human Review | Human (Knowledge Owner) | Review verdict per item || Human Review → Published | Human (decision) + CLI (materialization) | Published Knowledge in Project Knowledge |### §6.2 One Session = One Draft Package**One Architecture Session produces exactly one Draft Package.**This is non-negotiable. The rule exists to:- align review boundaries with reasoning boundaries (a Session has a coherent reasoning context; a Draft Package therefore has a coherent change context);- prevent partial Drafts from leaking into Project Knowledge before review;- bound the cognitive cost of Review (a Reviewer reviews one Draft Package, not an accumulating queue of partial changes);- give the Human a clean Publish/Reject decision unit.If a Session produces no new Knowledge worth preserving, it produces no Draft Package. If a Session's work cannot fit in one Draft Package (rare, but possible for very large architectural changes), the Session should be split before drafting — not the Draft Package.### §6.3 Repository Snapshot Philosophy**The Repository is reviewed as a Snapshot, not as a diff** (Decision-7).A Snapshot is the entire Repository at a point in time. The Review subject is the Snapshot itself.**Why not diff-based review:**- Knowledge integrity is a property of the *whole* Repository (Information Model, Schemas, Registry, Models, rendered Views all internally consistent). A diff cannot, in general, be reviewed for integrity.- A diff can mislead: a "small" diff may have large consistency implications elsewhere; a "large" diff may be a mechanical regeneration with zero integrity impact.- Review by Snapshot aligns the Human's mental model with what is being shipped: the Repository, not a patch.**CHANGELOG is for navigation, not review.**A Draft Package contains a CHANGELOG that lists what changed since the prior Snapshot. The CHANGELOG helps the Reviewer find the deltas. The CHANGELOG is **not** the review subject. A Reviewer who looks only at the CHANGELOG is not following PKMP's Review discipline.### §6.4 Draft Package StructureA Draft Package contains, at minimum:- `README.md` — what this Draft Package is and how to review it.- `CHANGELOG.md` — what changed since the prior Snapshot.- `REVIEW.md` — the review checklist: items the Reviewer must verify..- The complete **Repository Snapshot** — all files of the Repository at this point in time.The Repository Snapshot may live as the root of the Draft Package, or as a subdirectory named after the Snapshot (e.g., `snapshot/`). The exact placement is determined in §7 (Repository Architecture).**What a Draft Package is NOT:**- not a patch;- not only the changed files;- not a diff against a remote branch.**A Draft Package is delivered as a single artifact**: a directory or an archive (typically a `.zip`) containing the above. The delivery format makes it possible for the Human Reviewer to examine the Snapshot offline, without dependence on a specific version-control hosting platform.### §6.5 Conversation Handling**Conversation is not stored in the Repository.** This is a firm consequence of Decision-7, of "Conversation history SHALL NEVER be required for reconstruction" (PKMP_PROJECT_KNOWLEDGE_v1.0, Knowledge Preservation), and of Information Loss Zero (§5.6).**How Conversation feeds the Pipeline:**1. During a Session, the AI is *in* the Conversation.2. The AI identifies Knowledge Candidates — points where the Conversation has produced something worth preserving.3. The AI *Compiles* Candidates into Drafts. The Compilation preserves substance (§5.6); it does not transcribe the Conversation.4. The Conversation itself is not exported to the Repository.**What happens to the Conversation after the Session:**Nothing, from PKMP's perspective. The Conversation may persist in the AI tooling's own history (e.g., a chat platform's transcript), but PKMP does not depend on it. Any Knowledge worth keeping has been Compiled into the Draft Package.**The pending question of Knowledge Extractor** (Conversation → Knowledge Candidate as a formal Capability):Handover Part 4 proposes a **Knowledge Extractor** Capability that performs the Conversation → Knowledge Candidate transformation deterministically and at scale. The status was recorded as "正式採用は保留" (formal adoption pending). This document does not yet ratify the Knowledge Extractor; the decision is held for **§10.3**, where the Information Loss Zero architecture is detailed. Until then, the Conversation → Candidate transformation is performed by the AI as part of normal Chief Architect work.---## §7 — Repository Architecture### §7.1 Repository as ProjectionBy Decision-3, the Repository is a **generated projection** of the Information Model and Registry, not the source of truth. This section specifies the projection that PKMP normatively uses.**Implications of "Repository as Projection":**- The same Information Model + Registry could, in principle, project onto a different file layout and still be a valid PKMP Repository. The layout in §7.2 is the **normative default**: every PKMP Repository SHOULD use it unless the Registry explicitly overrides it.- The Repository's structural artifacts (top-level `README.md`, `PROJECT.md`, `ARCHITECTURE.md`, internal index files) are **generated**. They are Views (§3.6), not canonical content. A human edit to them does not change Knowledge.- A Repository can be **regenerated from scratch** at any time given the Information Model, Schemas, Models, Records, and Registry. The CLI's `regenerate` operation produces a byte-identical Repository (modulo non-deterministic metadata like timestamps).### §7.2 Top-Level StructureThe normative top-level structure of a PKMP Repository:```PKMP-Repository/├── README.md                — generated; entry point for any reader├── PROJECT.md               — generated; project identity (name, scope, owner)├── ARCHITECTURE.md          — generated; current architectural overview├── knowledge/               — META level: Information Model + Schemas│   ├── information-model/│   └── schemas/├── docs/                    — Documents (Knowledge Type)│   ├── <document>.yaml      — canonical Document Model│   └── <document>.md        — generated View (regenerable)├── records/                 — Records (Knowledge Type)│   ├── <record>.yaml        — canonical Record Model│   └── <record>.md          — generated View (regenerable)├── capabilities/            — Capabilities (Renderers, Validators, Extractors, …)│   ├── <capability>.yaml    — Capability Model (declaration / contract)│   └── <capability>/        — implementation directory (optional)├── views/                   — generated cross-cutting Views (HTML/PDF/OpenSpec/dashboards)├── registry/                — Repository Registry (self-describing metadata)└── provenance/              — original source documents, prior Handovers```**Refinement note** (resolving the ambiguity in Open Issue #2's recommendation):The Open Issue recommendation initially mentioned a `knowledge/models/` directory. On closer examination, putting **all** canonical Models in a single `knowledge/models/` bin would conflate Document Models, Record Models, Capability Models, and Registry Models — each of which has a distinct domain role. The refined structure above places canonical Models **next to their domain role** (`docs/`, `records/`, `capabilities/`, `registry/`) and reserves `knowledge/` strictly for the **meta level**: the Information Model and the Schemas that govern everything else. This is a clarification of the original recommendation, not a deviation: the original intent ("knowledge/ holds canonical things") is preserved at the meta level; the per-domain canonical Models are surfaced where they naturally belong.**`views/` vs. `docs/` vs. `records/`:**Note that `docs/` and `records/` each co-locate canonical YAML Models with their generated Markdown Views. This is deliberate: when a Reviewer opens `docs/`, they see both the canonical Knowledge and a rendered form they can read directly. The separate `views/` directory holds **cross-cutting generated outputs** that do not correspond one-to-one with a single Document or Record: aggregate HTML sites, PDF reports, OpenSpec output, dashboards, indexes.### §7.3 Sub-Directory Responsibilities#### §7.3.1 `knowledge/` — Meta LevelHolds the Knowledge that describes Knowledge.- `knowledge/information-model/` — the Information Model (§3.2).. The conceptual structure of Knowledge Types and relations for this Repository.- `knowledge/schemas/` — the Schemas (§3.3). Each Knowledge Type has at least one Schema file. JSON Schema-compatible YAML.The CLI MUST validate every Model in `docs/`, `records/`, `capabilities/`, and `registry/` against the relevant Schema in `knowledge/schemas/` before accepting it.#### §7.3.2 `docs/` — DocumentsHolds **Documents** (§3.8): human-facing deliverables that present conclusions.Each Document is a pair:- `docs/<doc-id>.yaml` — canonical Document Model.- `docs/<doc-id>.md` — generated Markdown View (regenerable).(Or: a per-document directory `docs/<doc-id>/` containing both, plus any attachments.)The Markdown file is a View (§3.6). Editing the Markdown does not change Knowledge. Regeneration overwrites Markdown edits.#### §7.3.3 `records/` — RecordsHolds **Records** (§3.8): the evidence trail, including Decision Records.Same pairing as `docs/`:- `records/<record-id>.yaml` — canonical Record Model.- `records/<record-id>.md` — generated Markdown View (regenerable).Records are **immutable once Published** (§3.8). The CLI MUST refuse in-place edits to a Published Record's YAML. A change to an existing Record's content requires a superseding Record (with cross-reference) and a Decision Record explaining the supersession.#### §7.3.4 `capabilities/` — CapabilitiesHolds **Capabilities** (§3.9): the executable components PKMP uses.For each Capability:- `capabilities/<capability-id>.yaml` — the Capability Model. Declares the Capability's contract, inputs, outputs, version, and (optionally) the OpenSpec reference.- `capabilities/<capability-id>/` — implementation directory (optional). Source code, scripts, binaries, or anything else the Capability needs to operate.The CLI MUST be able to enumerate available Capabilities from this directory alone, without consulting external registries.#### §7.3.5 `views/` — Cross-Cutting Generated ViewsHolds **cross-cutting generated outputs** that span multiple Documents/Records:- HTML site builds- PDF compilations- OpenSpec output- Dashboards (e.g., "all open Decision Candidates")- Aggregate indexesEverything in `views/` is regenerable from canonical Models. The CLI MUST be able to delete `views/` entirely and reconstruct it without loss.#### §7.3.6 `registry/` — Repository RegistryHolds the **Repository Registry** (§3.10, §8): metadata that describes the Repository itself.Registry files are Models too (they conform to Schemas in `knowledge/schemas/`). They are canonical: editing the Registry changes the Repository's projection. See §8 for the full Registry Specification.#### §7.3.7 `provenance/` — Original SourcesHolds the **original source documents** from which the current Repository was bootstrapped: the original handovers, the original Project Knowledge documents, any pre-PKMP design notes that informed the Repository.`provenance/` is **append-only**. The CLI MUST NOT overwrite or delete files here under any operation. Older provenance is never removed, only added to.This is how Decision-11's "discard provenance" prohibition is given a physical home.### §7.4 Repository Rule**The Repository Rule:**> Top-level directories hold **Repository-wide responsibilities**.> Project-specific content lives in **subdirectories**.This rule prevents top-level clutter as a project grows. For example: an SRM-specific Architecture Document goes in `docs/srm/architecture.md`, not `docs/srm-architecture.md` and certainly not `srm-architecture.md` at the root.**The Repository Rule itself is Knowledge.** It is recorded as a Record (specifically a meta-Decision Record). It is enforced by the CLI's validation pass.**Top-level whitelist.**Only the following top-level entries are permitted in a normative PKMP Repository:- `README.md`, `PROJECT.md`, `ARCHITECTURE.md` (generated entry files)- `knowledge/`, `docs/`, `records/`, `capabilities/`, `views/`, `registry/`, `provenance/`- A small set of conventional auxiliary files (`LICENSE`, `CONTRIBUTING.md`, `.gitignore`, `CHANGELOG.md`, `REVIEW.md` if the Repository is currently a Draft Package)Any top-level entry outside this whitelist constitutes a **Repository structure violation** and MUST be either moved to an appropriate subdirectory or sanctioned by a Decision Record.### §7.5 Repository RegistryThe Repository Registry lives in `registry/` and describes the Repository's own structure.Minimum Registry contents (full specification in §8):- `registry/repository.yaml` — Repository identity (name, version, owner, top-level structure)- `registry/documents.yaml` — index of Documents in `docs/`- `registry/records.yaml` — index of Records in `records/`- `registry/capabilities.yaml` — index of Capabilities in `capabilities/`- `registry/schemas.yaml` — index of Schemas in `knowledge/schemas/`- `registry/lifecycles.yaml` — definition of any project-specific lifecycles- `registry/workflows.yaml` — definition of any project-specific workflowsThe Registry is **canonical** — it is itself Knowledge, and changes to it are changes to the Repository's projection. The Registry is **not** auto-generated by scanning the filesystem: the filesystem reflects the Registry, not the other way around.### §7.6 Self-Describing RepositoryA consequence of §7.3–§7.5 is that a PKMP Repository is **self-describing**: any agent that gains read access to the Repository can determine, from the Repository alone:- what kinds of Knowledge it contains (from `knowledge/`);- what specific Knowledge items exist (from `registry/`);- where each item lives in the file layout (from `registry/` cross-referenced with `docs/`, `records/`, `capabilities/`);- what Capabilities are available to operate on the Knowledge (from `capabilities/`);- where the Repository came from (from `provenance/`).No external documentation, no out-of-band setup, no human briefing is required. This is the operational meaning of "Success is achieved when an independent CLI reconstructs PKMP from the published knowledge without needing prior chat history" (PKMP_PROJECT_KNOWLEDGE_v1.0, Success Criteria).The CLI MUST treat self-description as an invariant. Any operation that breaks self-description (e.g., adding files referenced nowhere in the Registry, or removing files referenced in the Registry) is a Repository structure violation.---## §8 — Registry Specification### §8.1 PurposeThe Registry exists to make the Repository **self-describing** (§7.6). It does so by recording, in canonical YAML, the answers to questions an agent might ask about the Repository:- "What Knowledge Types does this Repository recognize?"- "Where is the Document with id `arch-overview`?"- "What Capabilities can render this Knowledge Type?"- "What lifecycle does this Decision Record follow?"The Registry is **not** a search index, **not** a database, and **not** a cache. It is the canonical statement of the Repository's structure. The filesystem is a projection of the Registry, not the other way around.### §8.2 FormatRegistry files are **YAML**, for the same reasons Models are YAML (§3.4): machine-readable, schema-validatable, human-readable enough, AI-friendly.Every Registry file conforms to a Schema in `knowledge/schemas/`. Registry files that fail Schema validation are rejected by the CLI.### §8.3 Registry FilesThe minimum set of Registry files in a normative PKMP Repository:| File | Purpose ||---|---|| `registry/repository.yaml` | Repository identity, version, top-level structure declaration || `registry/documents.yaml` | Index of all Documents and their metadata || `registry/records.yaml` | Index of all Records, including Decision Records, with state || `registry/capabilities.yaml` | Index of all Capabilities, with version and contract reference || `registry/schemas.yaml` | Index of all Schemas in `knowledge/schemas/` || `registry/lifecycles.yaml` | Project-specific lifecycle definitions (extends the default Knowledge Lifecycle in §5.1) || `registry/workflows.yaml` | Project-specific workflow definitions (extends the default Publishing Pipeline in §6.1) |Additional Registry files MAY be added by a project for project-specific Knowledge Types, but the minimum set above MUST be present.### §8.4 Schema OutlineThe Schemas governing Registry files live in `knowledge/schemas/registry/`. This section specifies the **structure** only:```yaml# registry/repository.yaml — outlinerepository:  id: <string>            # Repository's stable ID  name: <string>          # human-readable name  version: <semver>       # Repository version (Knowledge Lifecycle scoped)  owner: <string>         # Knowledge Owner identifier  created: <date>  updated: <date>  top_level:              # the §7.2 normative whitelist; project-specific additions allowed    - knowledge/    - docs/    - records/    - capabilities/    - views/    - registry/    - provenance/``````yaml# registry/documents.yaml — outlinedocuments:  - id: <string>    title: <string>    schema: <ref to knowledge/schemas/.../*.yaml>    state: Draft | Reviewed | Published    canonical: docs/<doc-id>.yaml    views:      - docs/<doc-id>.md      - views/<doc-id>.html   # optional    depends_on:      - records/<record-id>.yaml   # provenance/justification``````yaml# registry/records.yaml — outlinerecords:  - id: <string>    type: DecisionRecord | ResearchRecord | ExperimentRecord | ...    schema: <ref>    state: Draft | Reviewed | Published    canonical: records/<record-id>.yaml    views:      - records/<record-id>.md    immutable: true   # Records are immutable once Published    supersedes: <record-id>   # optional: chain of supersession``````yaml# registry/capabilities.yaml — outlinecapabilities:  - id: <string>    type: Renderer | Validator | Extractor | Installer | Generator | ...    contract: <openspec ref or local Capability Model ref>    version: <semver>    canonical: capabilities/<capability-id>.yaml    implementation: capabilities/<capability-id>/   # optional```Full Schemas are produced by the CLI as part of Bootstrap Phase 0 (§11)..### §8.5 What the Registry ManagesThe Registry is the **single source of truth for Repository structure**. It governs:1. **Locator information**: where to find each piece of Knowledge.2. **Type information**: what Knowledge Type each item is.3. **State information**: where each item is in its Lifecycle (Draft / Reviewed / Published).4. **Capability bindings**: which Capabilities are available, and what they can do.5. **Lifecycle bindings**: which Lifecycle each Knowledge Type follows.6. **Workflow bindings**: which Workflows each Knowledge Type participates in.The Registry does **not** manage:- The content of any Document, Record, or Schema (those are in their own canonical files).- Version control history (that is Git's job, when Git is used).- Authentication or authorization (that is a deployment concern, outside PKMP Core).- Search indexes (those are generated from the Registry by a Capability, not stored in the Registry).---## §9 — Capability Model### §9.1 What a Capability IsA **Capability** is an executable component with a declared contract. The contract specifies inputs, outputs, side effects, and version. The contract is canonical; the implementation that satisfies the contract is exchangeable.Capabilities in PKMP:- have a **Capability Model** in `capabilities/<id>.yaml` (declaring contract, type, version);- MAY have an **implementation directory** in `capabilities/<id>/` (containing the code or scripts);- are **registered** in `registry/capabilities.yaml`;- are **invoked** by the CLI through the contract, not by direct call into implementation details.### §9.2 Why "Capability" Rather Than "Plugin"The term **Plugin** was considered and rejected. The reasoning, restated more fully than in §3.9:- **Plugin connotes a host-binding architecture.** A "plugin" implies a "host" that loads the plugin via a specific mechanism (dynamic linking, JS module loading, etc.). PKMP does not have a single host; the same Capability might be invoked by a CLI, a CI pipeline, an editor extension, or another Capability.- **Plugin connotes implementation, not contract.** When people say "the plugin," they usually mean the code. PKMP's primary concern is the contract: what the component promises to do, regardless of how.- **Plugin connotes a particular runtime.** A "plugin" is typically a piece of code in the same language as the host. A PKMP Capability might be a shell script, a Python program, a Rust binary, an external service, or a procedure executed by a human. The abstraction must accommodate all of these uniformly.**Capability** captures the contract-first, runtime-agnostic, host-agnostic intent.### §9.3 Capability ExamplesThe PKMP Capability vocabulary, with examples:#### RenderersConsume Models, produce Views.| Capability ID | Input | Output ||---|---|---|| `renderer.markdown` | any Document or Record Model | `.md` file || `renderer.html` | any Document or Record Model | `.html` file || `renderer.pdf` | any Document Model | `.pdf` file || `renderer.openspec` | Capability Model | OpenSpec YAML || `renderer.wiki` | any Document Model | wiki-flavored Markdown |#### ValidatorsConsume Models + Schemas, produce verdicts.| Capability ID | Purpose ||---|---|| `validator.schema` | Verifies a Model conforms to its declared Schema || `validator.cross-ref` | Verifies that all `depends_on` references resolve || `validator.registry` | Verifies the Registry matches the actual file layout || `validator.lifecycle` | Verifies that state transitions follow the declared Lifecycle |#### ExtractorsConsume Conversation (or other unstructured input), produce Knowledge Candidates.| Capability ID | Status ||---|---|| `extractor.knowledge` | **Pending** ratification — see §10.3 (Open Issue #3) |#### Installers / GeneratorsBootstrap-time Capabilities.| Capability ID | Purpose ||---|---|| `generator.bootstrap` | Generates a Phase-0 PKMP Repository from a hand-written Information Model + Schemas (see §11) || `generator.scaffold` | Scaffolds a new Document, Record, or Capability with template content conforming to the relevant Schema |#### External AgentsCapabilities that wrap external services or other AI agents.| Capability ID | Purpose ||---|---|| `agent.github` | Reads/writes a GitHub repository as a remote PKMP storage backend || `agent.ai-reviewer` | A separate AI instance that reviews a Draft Package and returns a review report |The list above is **illustrative**, not exhaustive, and not part of the bootstrap minimum. The bootstrap minimum is specified in §11.### §9.4 Exchangeability Principle**Capabilities are exchangeable.**The Information Model, the Schemas, the Registry, and the Models do not depend on any specific Capability implementation. If `renderer.markdown` is replaced with `renderer.markdown.v2` tomorrow, no canonical Knowledge changes — only the rendered Views update.**Operational rules:**1. **The CLI MUST NOT hardcode references to specific Capability implementations.** All Capability calls go through the contract declared in `capabilities/<id>.yaml`.2. **A Repository MUST function correctly with a minimal Capability set.** The minimum: `validator.schema`, `validator.registry`, `renderer.markdown`. If only these three Capabilities are available, the Repository's canonical content remains intact and its primary Markdown Views remain regenerable.3. **Adding a Capability is non-disruptive.** Adding `renderer.pdf` to a Repository adds PDF Views; it does not change anything else.4. **Removing a Capability degrades Views, not Knowledge.** Removing `renderer.pdf` removes the ability to regenerate PDF outputs but does not affect any canonical Knowledge.### §9.5 OpenSpec's PositionBy Decision-5, OpenSpec defines executable Capabilities. This section formalizes what that means in PKMP's architecture.**OpenSpec is a Capability description format**, not a PKMP-Core component:- OpenSpec MAY be used to describe a Capability's contract in detail (inputs, outputs, error conditions, versions).- OpenSpec descriptions, when present, are referenced from the Capability Model in `capabilities/<id>.yaml`.- OpenSpec descriptions live in `capabilities/<id>/openspec.yaml` (next to the implementation).**OpenSpec is NOT used for:**- Requirements management. PKMP requirements (if any project models them) are first-class PKMP Knowledge in `docs/` and `records/`, not OpenSpec definitions.- Document or Record schemas. Those use the Schema mechanism (§3.3) in `knowledge/schemas/`.- Repository structure or Registry definition. Those are PKMP-native.**The Repository does not depend on OpenSpec.** A PKMP Repository that uses no OpenSpec is fully valid. OpenSpec is a Capability description format that some Capabilities elect to use. Other Capabilities may describe their contracts in plain YAML conforming to a Capability Schema; that is equally valid.The CLI MUST treat the absence of OpenSpec as normal. The CLI MUST NOT require OpenSpec to be installed in order to perform basic Repository operations.---## §10 — Information Loss Zero Architecture### §10.1 Principle (Architectural Framing)§5.6 introduced **Information Loss Zero** as a discipline applied to Knowledge transitions. This section presents the architectural consequences of that discipline — the structural mechanisms by which Information Loss Zero is enforced rather than merely declared.**Structural enforcement points:**1. **Conversation never enters the Repository** (§6.5). Eliminates the most common loss path (summarized-conversation-becomes-truth).2. **Models are canonical; Views are derived** (Decision-4, §3.4–§3.6). A Markdown summary cannot displace its source Model.3. **Documents reference Records; Records are immutable once Published** (§3.8, §7.3.3). Rationale is preserved separately from conclusions and cannot be retroactively edited away.4. **Supersession, not deletion** (§5.2, §7.3.3). Changes produce new Knowledge that supersedes old; the old remains in history.5. **Provenance is append-only** (§7.3.7). Original sources cannot be discarded.6. **Snapshot review, not diff review** (Decision-7, §6.3). The whole Repository is the unit of review, so loss-by-omission is structurally harder to miss.Information Loss Zero is not a slogan; it is a property the Repository maintains because the structure makes alternatives expensive.### §10.2 Conversation Compilation**Compilation, not Summarization.**The transformation Conversation → Knowledge Candidate is named **Compilation**. The terminology distinguishes the operation from "summarization":| Aspect | Summarization | Compilation ||---|---|---|| Output size | Smaller than input | Whatever size faithful structure requires || Lossy? | Yes, by design | No; loss-zero is the contract || Discards detail? | Yes | No; detail is reshaped, not removed || Reversible? | Generally no | Conceptually yes: the Candidate carries enough to recover intent || Editorial judgment? | High (what to keep, what to drop) | Low (preserve everything; surface ambiguity rather than resolve silently) |**Compilation process (current, AI-driven):**1. **Identify Candidate units.** The AI scans the Conversation for points where a decision was made, a fact was established, an alternative was considered, an action was committed.2. **Shape each Candidate.** The AI structures each identified unit as a Knowledge Candidate (or Decision Candidate or Action Candidate) carrying the substance verbatim or near-verbatim plus the surrounding context required to interpret it.3. **Surface ambiguity.** Where the Conversation is unclear, the Candidate records the ambiguity as an Open Question or as multiple Decision Candidates, never silently resolved.4. **Cross-reference.** Each Candidate references the related Candidates from the same Session, building the Session's Candidate graph.5. **Compile into Draft Package.** All Candidates from a Session are bundled into a single Draft Package with the standard structure (§6.4).**The current Compilation is performed by the AI in the Chief Architect role..** It is a deliberate, transparent process the Human can audit Candidate-by-Candidate.**Whether to formalize Compilation as a Capability** (the Knowledge Extractor question) is the subject of §10.3, currently pending ratification (Open Issue #3).### §10.3 Knowledge Extractor CapabilityPer Decision Candidate **DC-002** (Appendix C), the Knowledge Extractor is **deferred** — neither ratified nor rejected.**Definition retained for forward reference.**A Knowledge Extractor, if ratified in future, would be a Capability with this contract shape:- **Input**: a Conversation transcript or other unstructured exploratory work product.- **Output**: one or more Knowledge Candidates conforming to the relevant Schema.- **Side effect**: provenance entries tying each Candidate to its source position in the input.The identifier `extractor.knowledge` is reserved for this future Capability. The CLI MUST NOT use this identifier for any other purpose.**Why deferred (summary; full Candidate text in Appendix C, DC-002).**The current Compilation path (AI in Chief Architect role performs Compilation manually per §10.2) works without the Capability. Adding the Capability now would commit PKMP to a contract shape before any implementation exists to test it against. This is precisely the risk Decision-11 exists to prevent (the CLI inventing behavior because the architecture allows a slot for it).**Review triggers (DC-002 will be revisited when):**1. Conversation Compilation becomes a measurable bottleneck for the AI-driven path; or2. A concrete Knowledge Extractor implementation is proposed for evaluation; or3. A non-Conversation extraction need arises (meeting transcripts, code-review threads, design documents, etc.) that would naturally fit this Capability slot.**What this means for the CLI today.**The CLI MUST NOT implement Knowledge Extractor behavior. The CLI MAY implement template generation for Knowledge Candidates as part of `generator.scaffold` (a different Capability — see §9.3), but this is **generation**, not **extraction**. The CLI MUST NOT consume Conversation transcripts directly.**What this means for the Repository today.**The Repository MAY include, under `provenance/`, transcripts or notes from past Architecture Sessions for human reference. These are not consumed by any automated process; they exist for human reading and for a possible future Knowledge Extractor implementation to potentially process.### §10.4 Provenance TrackingProvenance is the record of *where Knowledge came from*. PKMP tracks provenance at three levels.#### Level 1 — File-Level Provenance (`provenance/` directory)The `provenance/` directory holds the *original physical sources* from which the Repository was bootstrapped. For PKMP itself, this means:- `provenance/PKMP_PROJECT_KNOWLEDGE_v1.0.md` — the original Project Knowledge document.- `provenance/PKMP_CHAT_HANDOVER_v1.md` — the original Handover document (all four parts concatenated).- `provenance/PKMP_DESIGN_v1.md` — this design document, preserved as its own provenance.Per §7.3.7, `provenance/` is **append-only**. The CLI MUST NOT modify or delete any file in this directory under any operation. New provenance is added (e.g., when a subsequent design version is produced, the prior PKMP_DESIGN is preserved here), but old provenance is never removed.#### Level 2 — Model-Level Provenance (metadata in each Model)Every canonical Model carries a `provenance` block:```yamlprovenance:  source: <description of where this Knowledge originated>  derived_from:                     # optional: prior Models this supersedes or extends    - <model-id>  authored_by:    role: chief-architect           # one of: chief-architect, knowledge-owner, implementation-agent    identifier: <session-id or human-name or AI-model-identifier>  authored_at: <ISO-8601 date>  reviewed_at: <ISO-8601 date>      # set when state transitions to Reviewed  reviewed_by: <human-identifier>   # set when state transitions to Reviewed  published_at: <ISO-8601 date>     # set when state transitions to Published  published_by: <human-identifier>  # set when state transitions to Published```**Operational rules:**1. The CLI MUST NOT remove or alter a Published Model's `provenance` block.2. State transitions append fields to the `provenance` block; they do not modify existing fields.3. If a Model is superseded by a later Model, the superseding Model's `provenance.derived_from` references the prior Model. The prior Model's `provenance` is unchanged.#### Level 3 — Session-Level Provenance (Draft Package metadata)Each Draft Package carries metadata identifying the Session that produced it.. This lives in `session.yaml` at the Draft Package root:```yamlsession:  id: <session-id>                  # unique per Session  date: <ISO-8601 date>  ai_instance:    role: chief-architect    identifier: <AI-model-identifier-or-session-token>  human:    role: knowledge-owner    identifier: <human-name>  candidates_produced:              # summary of Candidate IDs surfaced this Session    - <candidate-id>  conversations_referenced:         # optional: paths under provenance/ if any    - <path>```#### Knowledge Extractor and ProvenanceWhen DC-002 is eventually ratified (if it is), the Knowledge Extractor's output will populate `provenance.source: extracted-from-conversation` and `provenance.authored_by.role: extractor` with the extractor's identifier. The Level 2 `provenance` block schema already accommodates this — adding the new role value `extractor` is the only schema change required at that point.Until then, all Compilation output carries `provenance.authored_by.role: chief-architect` because the AI in Chief Architect role performs it.---## §11 — Bootstrap Plan### §11.1 The Bootstrap ProblemPKMP is a Knowledge Modeling Platform. Future contributors will be tempted to re-propose them; this section records why they should not be reintroduced without strong new reasoning.**Project Operating System (POS).** The original project name. Rejected because "Operating System" implies runtime orchestration of processes, which PKMP does not do. PKMP models Knowledge; it does not orchestrate operations. POS is preserved as the name of a *possible future Capability* under PKMP (an operational/runtime-management Capability), not as the platform itself.**Kernel.** Considered as a name for the PKMP Core. Rejected because "Kernel" implies a privileged-core + user-space architecture that PKMP does not have. PKMP's Information Model, Schemas, and Registry are canonical content, not privileged executable code.**Plugin.** Considered as the term for Capability. Rejected for the reasons in §9.2 — host-binding, runtime-specific, implementation-implying connotations.**Abolishing `docs/`.** Considered: route all human-facing reading through generated `views/` only, with no `docs/` top-level directory. Rejected because the Document Knowledge Type (§3.8) is architecturally distinct from cross-cutting Views, and surfacing this distinction at the top level helps human navigation. `docs/` and `views/` serve different purposes.**PROJECT.yaml-only operation.** Considered: have the Repository consist entirely of a single PROJECT.yaml plus generated everything. Rejected because Document/Record separation, Capability layering, and Registry distinctness need their own structural presence; a single file cannot adequately represent the architecture.**Massive monolithic Markdown Repository.** Considered: treat the Repository as a flat collection of large Markdown files (typical Obsidian or Logseq style). Rejected because it would make Markdown canonical (violating Decision-4), would prevent Schema-driven validation, and would obstruct the Document/Record separation.**`apiVersion: pkmp.io/v1alpha1` Kubernetes-style metadata.** Encountered in `PKMP_Repository_Snapshot_0004.zip` during the drafting of this document. Rejected. This identifier appears nowhere in either source document and represents the exact kind of unauthorized concept-invention that Decision-11 forbids. PKMP's identifiers are PKMP-specific and follow §7 and §8, not Kubernetes or other unrelated systems.**Direct Conversation storage.** Considered: include the full Conversation transcript in the Repository so that future agents can re-derive Knowledge. Rejected because (a) the AI's job is to Compile Conversation into Knowledge (Information Loss Zero, §5.6); (b) "Conversation history SHALL NEVER be required for reconstruction" is explicit (PKMP_PROJECT_KNOWLEDGE_v1.0, Knowledge Preservation); (c) storing Conversation would invite agents to depend on it rather than on canonical Knowledge.**Bidirectional Renderer (View → Model parsing).** Considered: have Renderers also parse edited Markdown back into Model updates. Rejected because it makes Markdown a write-canonical surface (violating Decision-4) and creates path-of-least-resistance for editing Views rather than Models. The View-to-Model direction, where required, is the job of an Extractor Capability (currently deferred, DC-002), not of a Renderer.---## §13 — Open QuestionsThese questions are not resolved by this design document. End every Session by producing exactly one Draft Package (§6.2). If a Session produces no Knowledge worth preserving, produce no Draft Package.### §14.2 Do's and Don'ts#### The CLI MUST:1. Maintain the Self-Describing Repository invariant (§7.6).10. Stop and surface the situation when ambiguity is encountered, per §14.3.#### The CLI MUST NOT:1. Introduce new top-level directories outside the §7.4 whitelist without a Decision Record sanctioning the addition.5. Introduce new architectural concepts not present in this document (Decision-11).6. Silently rename canonical identifiers (Decision-11).7. Publish unilaterally (Decision-10; only the Human Knowledge Owner Publishes).8. Consume Conversation transcripts directly (§10.3, §14.4).9. Require OpenSpec for basic operations (§9.5).10. Implement Knowledge Extractor behavior (§10.3, DC-002).11. Treat the prior `PKMP_Repository_Snapshot_0004.zip` artifact as authoritative for any structural decision (§2.11, §12).### §14.3 Handling AmbiguityWhen the CLI encounters ambiguity in the design:1. **Consult the sources, in priority order.**   - This document (§0–§14, Appendices).   - `provenance/PKMP_PROJECT_KNOWLEDGE_v1.0.md`.   - `provenance/PKMP_CHAT_HANDOVER_v1.md`.   - Stop at the first answer found.2. **Consult Decision Records and Decision Candidates** (Appendix C and `records/`). They may resolve the ambiguity.3. **If still ambiguous, surface the gap.** Produce a Decision Candidate proposing one or more resolutions. Do not guess.4. **Default to conservatism.** Where the design says one thing and a possibly-more-helpful interpretation exists, follow what the design says. The CLI's preferences are not authority.#### The CLI MUST NOT:- Silently resolve ambiguity by picking an interpretation.- Assume omission means permission ("the design doesn't forbid it, so I may do it").- Treat its own training-data prior as authoritative.### §14.4 Knowledge Candidate Extraction RulesThis subsection covers the case where the CLI processes a Session whose AI did not explicitly identify every Candidate. The CLI MAY assist Candidate identification, subject to these rules:1. **Candidate identification is the AI's responsibility, not the CLI's.** The AI in Chief Architect role identifies Candidates as part of normal work. The CLI MAY *suggest* additional Candidates the AI might have missed, but the CLI MUST NOT modify the AI's Candidate set without the AI's explicit confirmation.2. **Knowledge Extractor remains unratified** (DC-002, §10.3). The CLI MUST NOT invoke any "Knowledge Extractor" Capability until DC-002 is ratified..3. **When in doubt, surface as Open Question.** A piece of Session content that might or might not be a Candidate is recorded as an Open Question in the Draft Package's REVIEW.md, not silently included or excluded.4. **No automated Conversation parsing.** The CLI MUST NOT parse Conversation transcripts directly; that is the AI's role. The CLI MAY read transcripts stored under `provenance/` for human-orientation purposes (e.g., when generating a session.yaml that references prior transcripts) but MUST NOT extract Knowledge from them.5. **Template scaffolding is permitted.** The CLI MAY scaffold empty Knowledge Candidate, Decision Candidate, and Action Candidate templates conforming to the relevant Schemas, for the AI or Human to populate. This is `generator.scaffold`-class work and is distinct from extraction.---## Appendix A — GlossarySingle-source definitions of every load-bearing term in this document. Sorted alphabetically.
