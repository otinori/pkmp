## Context

PKMP projects are initialized by copying dist/ into the target repository and
running pkmp-init. The dist/registry/ directory contains starter versions of all
registry files. During Simulation Round 7 (Sim 39), a systematic diff between
dist/registry/ and registry/ revealed that lifecycles.yaml, workflows.yaml,
schemas.yaml, and capabilities.yaml all contained only empty lists in dist/.


## Decision

Always populate dist/registry/ with the PKMP built-in definitions before cutting
a distribution release:
- lifecycles.yaml: document, record, spec lifecycle state machines
- workflows.yaml: all PKMP workflow definitions (process-inbox through apply-process-improvement)
- schemas.yaml: all built-in schema paths pointing to .pkmp/schemas/
- capabilities.yaml: built-in provider defaults (UDR, Conclave fallbacks)
Only documents.yaml, records.yaml, and repository.yaml should remain as
empty/placeholder stubs (they are project-specific).


## Rationale

Empty lists in dist/registry/ cause any project initialized via pkmp-init to run
with no lifecycle rules, no workflow definitions, no schema mappings, and no
capability routing. All PKMP skills silently degrade: pkmp-verify cannot check
actor constraints, spec-assist cannot route to the right provider, and records
will fail schema validation because schemas.yaml maps no schemas.


## Learning

The dist/ template is the single source of truth for what new projects inherit.
Built-in definitions (lifecycles, workflows, schemas, capabilities) are framework
defaults, not project-specific data — they belong in dist/. Failing to sync dist/
after adding new lifecycle states, workflow steps, or schemas creates a silent
divergence: the PKMP runtime (registry/) knows the rules but new projects start
without them.


## Applicability

Applies whenever: (a) cutting a PKMP distribution release, (b) adding a new
lifecycle or workflow, (c) adding a new built-in schema, (d) adding a new
built-in capability. Always update dist/registry/ in sync with registry/ for
the four built-in files.
