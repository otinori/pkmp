## Context

During simulation Round 6, we traced a scenario where a SPEC document contains
a reference to LR-001 in its Constraints section. Later, LR-001 is superseded
by a new decision (DC-005). The SPEC now carries a stale constraint based on
outdated knowledge, but pkmp-verify only validates YAML schema correctness —
it does not check whether referenced record ids are still current (Published).


## Decision

Extend pkmp-verify to include a stale-reference check:
1. For each document canonical YAML in docs/, scan the `content` field for
   strings matching known record ids (LR-NNN, DC-NNN, etc.)
2. Look up each referenced id in registry/records.yaml
3. If state is Superseded, emit a warning (not a hard error):
   "WARN: docs/spec-auth.yaml references DC-003 which is Superseded by DC-007.
    Consider updating the document."
This should be a warning rather than a hard error because:
- The SPEC may intentionally reference a superseded decision for historical context
- Hard errors would block commits for existing SPECs


## Rationale

Without this check, outdated constraints can silently persist in SPECs,
misleading future implementors who assume all SPEC constraints are current.
A warning surfaces the issue without blocking work.
