# [PI-008] Record schema should not require decision/rationale for non-DecisionRecord types

## Context

The current record.schema.yaml requires `decision` and `rationale` for ALL record
types (they are in the top-level `required` array). However, these fields are only
semantically meaningful for DecisionRecord. LearningRecord types use `learning` and
`applicability` instead. ResearchRecord, ExperimentRecord, etc. have their own
semantic fields.

Currently, LearningRecords (LR-001, LR-002) work around this by repurposing
`decision` (as action recommendation) and `rationale` (as applicability). This
causes conceptual confusion: LR records have both `decision`+`rationale` AND
`learning`+`applicability`, with overlapping/redundant content.


## Decision

Update record.schema.yaml to use conditional validation:
- `decision` and `rationale` remain in `required` only when `type: DecisionRecord`
- All other record types require only: id, type, title, context, provenance
- `learning` and `applicability` become required for `type: LearningRecord`

JSON Schema draft-07 `if/then` syntax:
```yaml
required:
  - id
  - type
  - title
  - context
  - provenance

if:
  properties:
    type:
      const: DecisionRecord
then:
  required:
    - decision
    - rationale
```

After schema update:
- Remove redundant `decision` and `rationale` from existing LR records (LR-001, LR-002)
- Verify pkmp-verify passes for all records


## Rationale

Field name mismatch causes confusion when reading LearningRecords — the `decision`
field in an LR record is not a "decision" but rather a recommended action. Using
wrong field names degrades knowledge retrieval quality because the semantic meaning
of each field guides how spec-assist extracts and applies the knowledge.
Conditional schema validation allows record types to be semantically clean without
requiring shared boilerplate fields.
