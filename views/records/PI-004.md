## Context

When capture-decision is used with UDR (Path A), it creates a cross-reference entry
in registry/records.yaml with schema: external/udr and canonical: .udr/records/<id>.yaml.
The pkmp-verify validator (schema.js) attempts to:
1. Load the canonical file (.udr/records/<id>.yaml) — succeeds (UDR creates this)
2. Load the schema file (external/udr) — FAILS: "Schema file not found: external/udr"

Additionally, even if the schema existed, UDR's YAML format is different from PKMP's
record schema, so validation would fail anyway.

Result: every UDR decision captured via Path A causes pkmp-verify to fail, blocking
commits for projects that use UDR.


## Decision

Two-part fix:

Part 1 — Validator update (.pkmp/lib/validators/schema.js):
Skip schema validation when the registry entry's `schema` field starts with "external/".
Add a comment to the output: "[SKIP] external/udr — external schema, validation skipped."

Part 2 — Validator update: Also skip canonical file validation when schema is external,
since the canonical format (UDR YAML) is not governed by PKMP's schema system.

Code change:
  // In validateAllSchemas, records loop:
  if (rec.schema.startsWith('external/')) {
    // Skip validation for externally managed records
    return;
  }


## Rationale

Projects using UDR for decision management need pkmp-verify to pass so they can commit.
External records should be trusted as valid by the external tool that owns them.
This is consistent with how cross-references work: PKMP records their existence,
the external tool ensures their correctness.
